## Tech Stack

### Frontend
- **Next.js** (App Router) + **React**
- **TypeScript**
- **Tailwind CSS**
- **Zustand** вЂ” client state (e.g. booking modal store)
- **lucide-react** вЂ” icons
- **Supabase JS client** вЂ” frontend Supabase access

### Backend
- **Node.js** + **Express**
- **TypeScript**
- **Prisma ORM**
- **PostgreSQL**, hosted on **Supabase**
- **Multer** (or equivalent) вЂ” file uploads (payment screenshots, images)

### Infrastructure
- **Frontend deployment:** [Vercel](https://vercel.com)
- **Backend deployment:** [Render](https://render.com) (free tier)
- **Database:** Supabase Postgres, accessed via Prisma (pooled connection for the app, direct connection for migrations)
- **Uptime monitoring:** [UptimeRobot](https://uptimerobot.com) pings `https://maan-travels-backend.onrender.com/health` every 5 minutes to keep the free Render instance warm and catch downtime

## Getting Started

### Prerequisites
- Node.js (LTS recommended)
- npm
- A PostgreSQL database (this project uses Supabase)

### 1. Clone the repository

```bash
git clone <repository-url>
cd <repository-folder>
```

### 2. Backend setup

```bash
cd backend
npm install
```

Create a `.env` file in `backend/` (see [Environment Variables](#environment-variables) below).

Generate the Prisma client and apply the schema:

```bash
npx prisma generate
npx prisma migrate deploy   # or: npx prisma db push, depending on your workflow
```

Run the backend in development:

```bash
npm run dev
```

The API will be available at `http://localhost:5000` (or whatever `PORT` you set).

### 3. Frontend setup

```bash
cd frontend
npm install
```

Create a `.env.local` file in `frontend/` (see [Environment Variables](#environment-variables) below).

Run the frontend in development:

```bash
npm run dev
```

The site will be available at `http://localhost:3000`.

## Environment Variables

### Backend (`backend/.env`)

| Variable | Description |
|---|---|
| `DATABASE_URL` | Pooled (transaction-mode, PgBouncer) Postgres connection string вЂ” used by the running app |
| `DIRECT_URL` | Direct (session-mode) Postgres connection string вЂ” used by Prisma for migrations |
| `SUPABASE_URL` | Supabase project URL |
| `SUPABASE_ANON_KEY` | Supabase anonymous/public key |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase service role key (server-side only вЂ” full access, keep secret) |
| `DEVICE_TOKEN_SECRET` | Secret used for signing/verifying admin device tokens |
| `FRONTEND_URL` | Frontend origin, added to the CORS allowlist |
| `PORT` | Port the Express server listens on (defaults to `5000`) |

### Frontend (`frontend/.env.local`)

| Variable | Description |
|---|---|
| `NEXT_PUBLIC_BACKEND_URL` | Base URL of the backend API |
| `NEXT_PUBLIC_API_URL` | Base URL used by frontend service calls to the API |
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase project URL (public) |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase anonymous/public key (public) |

### CORS

The backend explicitly whitelists origins rather than using a wildcard, since `credentials: true` (required for the admin device cookie) is rejected by browsers when combined with `origin: "*"`. Allowed origins:
- `https://www.maantravels.com`
- `http://localhost:3000`
- `process.env.FRONTEND_URL`
- any `*.vercel.app` subdomain (preview deployments)

## Database Schema

Defined in `backend/schema/schema.prisma`, using PostgreSQL via Prisma.

### Core models

- **`Customer`** вЂ” name, email, phone; has many `Booking`s
- **`Booking`** вЂ” the central record for every booking, regardless of service type. Holds `serviceType`, `status`, pricing (`totalAmount`, `paymentType`, `amountPaid`, `paymentStatus`), and an optional link to a `Vehicle` and/or `Route`. Each `Booking` has *one* of the following service-specific child records:
  - **`TaxiBooking`**
  - **`AirportTransfer`**
  - **`TourBooking`**
  - **`SelfDriveBooking`**
  - **`LuxuryBooking`**
  - **`WeddingBooking`**
  - **`TempoBooking`**

- **`Vehicle`** вЂ” fleet vehicles. Shared across taxi, self-drive, and tempo bookings via flags:
  - `isTaxiFleet` (default `true`) and `isSelfDrive` (default `false`) вЂ” independent booleans; a vehicle **can be both** (e.g. an Innova Crysta available for taxi trips *and* self-drive rental)
  - `category` (free text, e.g. `"luxury"`, `"suv"`, `"sedan"`) вЂ” luxury vehicles are identified by `category` containing `"luxury"`, not a separate boolean
  - Self-drive-specific fields: `fuelType`, `transmission`, `seats`, `modelYear`, `rentalPerDay`
  - General pricing: `price` (used for taxi/tour context); `rentalPerDay` (used for self-drive context)

- **`LuxuryCar`** вЂ” catalog entries for luxury vehicles (slug, images, features), each linked 1:1 to a `Vehicle`

- **`Route`** вЂ” predefined route cards (e.g. "Popular Tour Routes", "Popular Destinations"), categorized by `category` (`destination` | `one_way` | `local` | `airport` | `tour`) and `baseType` (`taxi` | `airport` | `tour`)
- **`RoutePricing`** вЂ” per-vehicle price for a given `Route` (unique on `routeId` + `vehicleId`)

- **`Airport`** / **`AirportPricing`** вЂ” airport catalog and per-vehicle airport transfer pricing, mirroring the `Route`/`RoutePricing` pattern

- **`TourLocation`** вЂ” pickup/drop location options (`canPickup`, `canDrop` flags)

- **`PaymentSettings`** вЂ” stores the QR code image and UPI ID shown during checkout
- **`Feedback`** вЂ” customer testimonials, with a `showOnWebsite` flag for moderation
- **`GalleryImage`** вЂ” gallery photos, categorized and orderable
- **`Inquiry`** вЂ” contact form submissions (`status`: `new` | `contacted` | `closed`)

- **`AdminDevice`** / **`AdminDeviceInvite`** вЂ” trusted-device authentication for `/admin`. A device is `pending` until approved via an invite code created by an already-approved device, after which its `status` becomes `approved` (or `revoked` if access is later pulled). The `active` boolean is kept in sync with `status === "approved"` for backward compatibility вЂ” it should not be written to independently of `status`.

### Key relationships to know

- A `Vehicle` with both `isTaxiFleet: true` and `isSelfDrive: true` is **not an edge case to special-case away** вЂ” it's valid data, and the frontend resolves which price/label to show based on which filter context (taxi vs. self-drive) the vehicle is currently being viewed under.
- `RoutePricing`/`AirportPricing` are the *only* source of truth for route-specific or airport-specific pricing. There's no fallback to `Vehicle.price` for these вЂ” if no pricing row exists, the booking is created with `totalAmount: 0` and the team follows up manually to confirm pricing.

## Deployment

| Layer | Platform | Notes |
|---|---|---|
| Frontend | Vercel | Auto-deploys from the connected branch; preview deployments get a `*.vercel.app` URL, which is allow-listed in backend CORS |
| Backend | Render (free tier) | Free tier spins down on inactivity вЂ” see Uptime Monitoring below |
| Database | Supabase (Postgres) | Accessed via Prisma; pooled connection (`DATABASE_URL`) for the app, direct connection (`DIRECT_URL`) for migrations |

### Uptime Monitoring

Render's free tier spins the backend down after a period of inactivity, causing a slow "cold start" on the next request. **UptimeRobot** pings [`https://maan-travels-backend.onrender.com/health`](https://maan-travels-backend.onrender.com/health) every 5 minutes to keep the instance warm and to provide independent alerting if the backend ever goes down.
